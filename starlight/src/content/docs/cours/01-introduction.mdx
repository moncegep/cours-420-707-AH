Dans ce premier chapitre, nous installons le modèle mental du développement côté serveur avant toute syntaxe : qui fait quoi entre le navigateur et le serveur, ce qu'apporte PHP, et comment une page se construit à chaque requête.

## Client et serveur

Quand vous accédez à une page web, le navigateur n'a pas le contenu d'avance. Il envoie une **demande** au serveur associé à l'adresse tapée dans la barre de navigation, par exemple `https://www.collegeahuntsic.qc.ca`. C'est pour cette raison qu'on l'appelle le <Highlight>client</Highlight> : il demande, il ne produit pas le contenu.

<MarginNote type="rappel">
Le navigateur n'est pas le seul client possible. Une application mobile, un autre serveur, un objet connecté (ex: voiture, réfrigérateur, borne) ou un simple script peuvent aussi jouer ce rôle : tout ce qui envoie une requête est un client.
</MarginNote>

Le serveur, lui, reçoit cette demande, **exécute le code** associé à l'adresse, et renvoie une **réponse**.

## Une petite histoire de PHP

PHP est né en 1994-1995. Rasmus Lerdorf l'a d'abord écrit comme un ensemble de scripts pour suivre les visites de sa page personnelle en ligne ; le nom venait de « Personal Home Page ». Le besoin initial était concret : <Highlight couleur="ambre">rendre les pages web dynamiques et traiter les données de formulaires</Highlight> côté serveur, sans réécrire un programme complet pour chaque page.

### L'écosystème

PHP fait tourner une grande partie du web : selon W3Techs, environ **75 %** des sites dont on connaît le langage serveur l'utilisent, et WordPress (écrit en PHP) propulse à lui seul plus de **40 %** des sites. Autour du langage gravitent :

- **Composer** : le gestionnaire de dépendances (l'équivalent de npm pour PHP).
- **Des frameworks** : Laravel et Symfony pour les applications, en plus de plateformes prêtes à l'emploi comme WordPress, Drupal et Magento.

## Page statique vs page dynamique

Une page dont **tout le contenu est déjà en HTML sur le serveur** est une page **statique** : le serveur la renvoie telle quelle, identique pour tout le monde.

<MarginNote type="rappel">
Du contenu dynamique peut tout de même être ajouté ensuite, côté client, avec du JavaScript. Mais le fichier servi par le serveur, lui, reste le même.
</MarginNote>

À l'inverse, une page **dynamique** <Highlight>crée son contenu au moment où le serveur reçoit la requête</Highlight>. 
Le contenu peut alors varier et s'adapter automatiquement : une page de produits générée depuis une base de données, un tableau de bord qui change selon l'utilisateur connecté, ou simplement une page qui affiche la date du jour.

{/* | Aspect | Page statique | Page dynamique |  
| --- | --- | --- |  
| Contenu | Fixe, écrit à l'avance | Généré à l'exécution |  
| Exemple | Un fichier `.html` | Une page `.php` qui lit une base de données |  
| Varie selon le contexte | Non | Oui (utilisateur, données, heure...) |  
| Rôle du serveur | Renvoyer le fichier | Exécuter du code, puis renvoyer le résultat |   */}

<Aside type="exemple" title="Le même besoin, deux approches">
Pour afficher une liste de produits : en statique, on écrit chaque produit à la main dans le HTML et on modifie le fichier à chaque changement. En dynamique, on stocke les produits une fois (dans un tableau ou une base de données) et la page les affiche automatiquement, à jour à chaque visite.
</Aside>

{/* <Checkpoint question="Une page qui affiche « Bonjour » suivi du prénom de l'utilisateur connecté est-elle statique ou dynamique ?">
Dynamique : son contenu dépend de qui consulte la page, donc il est produit à l'exécution, à la réception de la requête.
</Checkpoint> */}

## Le cycle d'une requête

Quand une page PHP est demandée, la requête traverse trois acteurs avant que le navigateur n'affiche le résultat.

<Steps>

1. Le **navigateur** envoie une requête HTTP (par exemple `GET /index.php`).

2. Le **serveur** (Apache) repère le fichier et le confie à l'**interpréteur PHP**.

3. L'interpréteur PHP **exécute** le code et produit du texte, généralement du HTML.

4. Le serveur renvoie ce HTML ; le navigateur l'affiche. Le code source visible ne contient **aucune** instruction PHP.

</Steps>



## XAMPP, votre environnement de travail

Pour exécuter du PHP sur votre machine, il faut trois choses : un **serveur web** qui reçoit les requêtes, l'**interpréteur PHP** qui exécute le code, et une **base de données** pour stocker les données. **XAMPP** réunit le tout en une seule installation :

- **Apache** : le serveur web ;
- **PHP** : l'interpréteur ;
- **MariaDB / MySQL** : la base de données ;
- **phpMyAdmin** : une interface web pour gérer la base de données.

<Aside type="prerequis" title="Avant de coder">
Installez XAMPP et démarrez **Apache** depuis son panneau de contrôle. Vos fichiers `.php` doivent être placés dans le dossier `htdocs`, la racine web d'Apache.
</Aside>

Le flux de travail est toujours le même :

<Steps>

1. Démarrez **Apache** dans le panneau de contrôle XAMPP.

2. Placez votre fichier dans `htdocs` (par exemple `htdocs/essai.php`).

3. Ouvrez `http://localhost/essai.php` dans le navigateur.

</Steps>

{/* <Aside type="erreur" title="Erreur fréquente">
Si vous voyez le code PHP s'afficher au lieu de la page, c'est que le fichier n'a pas été servi par Apache. Ouvrez-le toujours via `http://localhost/...`, jamais en double-cliquant le fichier (`file://`).
</Aside> */}

## Récapitulatif

- Le **client** (navigateur ou autre) envoie une requête ; le **serveur** exécute du code et renvoie une réponse.
- **PHP s'exécute côté serveur** et ne renvoie que le HTML produit, jamais son propre code.
- Une page **statique** est servie telle quelle ; une page **dynamique** est construite à l'exécution et peut s'adapter.
- Le **cycle d'une requête** : navigateur, puis Apache, puis interpréteur PHP, puis retour du HTML.
- **XAMPP** fournit Apache, PHP et MySQL pour développer localement.
